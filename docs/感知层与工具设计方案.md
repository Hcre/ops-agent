# 感知层与工具设计方案

## 方案定位

这是一个**面向运维场景的分层 Agent 系统**，核心原则是：**不把全量数据直接给 LLM，而是先构造可定位、可追查、可按需展开的上下文。**

---

## 总体架构（四层设计）

```
采集层 → 感知层 → 上下文工程层 → LLM 层
```

| 层级 | 职责 | 输出 |
|------|------|------|
| **采集层** | 采集主机、进程、磁盘、网卡、日志、变更、拓扑等原始数据 | 原始指标和事件 |
| **感知层** | 基线对比、异常检测、实体识别 | 对象概况包 + 异常标识 |
| **上下文工程层** | 优先级排序、摘要压缩、token 预算控制 | 注入 LLM 的上下文 |
| **LLM 层** | 归纳、推理、诊断，按需调用工具 | 诊断结论和建议 |

---

## 感知层设计

### 职责

感知层不负责把问题讲完，而是回答三个问题：

1. **是什么对象出问题了**
2. **出了什么异常**
3. **下一步该查什么工具**

### 输出格式：概况包

每个异常对象输出一个"概况包"，包含：

```json
{
  "entity_type": "process",           // 实体类型
  "entity_id": "process-3381",        // 实体标识
  "host_id": "host-12",               // 所属主机
  "severity": "HIGH",                 // 异常级别：CRITICAL/HIGH/MEDIUM/LOW
  "summary": "该进程在 10 分钟内重启 4 次，内存持续上涨，疑似 OOM",
  "time_window": "15:00 - 15:10",     // 时间窗口
  "evidence_pointers": [              // 关联证据指针
    "log-group-77",
    "host-12:dmesg",
    "change-20260416-01"
  ],
  "suggested_tools": [                // 推荐工具
    "get_process_detail",
    "get_logs",
    "get_change_detail"
  ],
  "suggested_mode": "detail",         // 推荐查询模式
  "confidence": 0.85,                 // 置信度
  "next_hints": [                     // 下一跳线索
    "process-3381",
    "log-group-77"
  ]
}
```

### 概况包的三个关键特性

1. **可定位**：每个异常都有唯一的实体 ID
2. **可追查**：包含证据指针和推荐工具
3. **可按需展开**：LLM 可以通过工具继续深入

---

## 工具设计原则

### 基本原则

- **按对象专用**：不做万能工具，而是为每类对象设计专用工具
- **统一接口**：所有工具支持相同的参数和输出格式
- **分层输出**：支持 `summary | detail | full` 三个级别

### 工具列表

| 对象 | 工具 | 参数 | 返回重点 |
|------|------|------|---------|
| 主机 | `get_host_detail(host_id, mode)` | host_id, mode, time_window | CPU、内存、磁盘、负载、异常摘要 |
| 进程 | `get_process_detail(pid, mode)` | pid, mode, time_window | PID、命令行、资源占用、重启、父子关系 |
| 磁盘 | `get_disk_detail(mount_id, mode)` | mount_id, mode, time_window | 使用率、I/O、Top 写入进程 |
| 网卡 | `get_interface_detail(iface_id, mode)` | iface_id, mode, time_window | 速率、丢包、错误包、状态 |
| 日志 | `get_logs(group_id, mode)` | group_id, mode, time_window | 错误段、stacktrace、Caused by 链 |
| 变更 | `get_change_detail(change_id, mode)` | change_id, mode | 变更类型、时间、对象、diff |
| 拓扑 | `get_topology_detail(service_id, mode)` | service_id, mode | 上下游关系、关键依赖、影响面 |

### 工具输出协议（统一格式）

```json
{
  "entity": {
    "type": "process",
    "id": "process-3381"
  },
  "mode": "summary",
  "time_window": "15:00 - 15:10",
  "summary": "进程状态摘要",
  "key_evidence": [
    "CPU 占用 85%",
    "内存占用 2.3GB（上升 500MB）",
    "重启次数 4 次"
  ],
  "related_entities": [
    "host-12",
    "log-group-77"
  ],
  "suggested_next_mode": "detail",
  "suggested_tools": [
    "get_logs",
    "get_host_detail"
  ]
}
```

## 工具输出策略

### summary（默认）

给 LLM 的默认结果，包含：

- 当前状态
- Top 异常项（3-5 个）
- 差值或趋势（而非绝对值）
- 简短解释

**示例**：
```
进程 process-3381：
- 状态：异常
- CPU：85%（↑ 30% vs 5min ago）
- 内存：2.3GB（↑ 500MB vs 5min ago）
- 重启：4 次（过去 10 分钟）
- 疑似原因：OOM
```

### detail

补充更多上下文：

- 更多历史数据
- 邻近对象
- 关联日志片段
- 子树或依赖项

**示例**：
```
进程树：
- parent: systemd (PID 1)
- children: 3 个子进程
- 相关日志：
  - 15:05 OOM killer 触发
  - 15:06 进程重启
  - 15:07 内存再次上升
```

### full

原始详情，仅在明确需要时返回：

- 完整日志窗
- 全量进程列表
- 全量历史采样
- 原始事件流

**限制**：
- 只允许短时间窗口
- 只允许特定对象
- 需要显式确认后调用

## 数据提炼规则

### 核心提炼原则

| 规则 | 说明 | 示例 |
|------|------|------|
| **Top3-5** | 只保留最异常的对象 | 磁盘异常中只看 Top 3 个写入进程 |
| **差值** | 突出变化量，不堆绝对值 | "1 分钟前到现在增长 500MB"而非"当前 2.3GB" |
| **窗口化** | 只看异常前后固定时间段 | 看异常前后 5-15 分钟，不看全天历史 |
| **模式识别** | 识别常见异常模式 | OOM、stacktrace、重启风暴、连接耗尽 |
| **实体归一** | 统一映射到稳定 ID | 目录、挂载点、进程、网卡都有唯一 ID |

### 具体提炼方法

1. **进程异常**
   - 只看：CPU、内存、重启次数、命令行、父进程、端口、最近错误
   - 不看：全量 ps 输出、历史采样

2. **磁盘异常**
   - 只看：挂载点、使用率、读写速率、Top 写入进程
   - 不看：全量 iostat 历史

3. **网卡异常**
   - 只看：当前速率、丢包、错误包、接口状态、差值
   - 不看：全量 ethtool 输出

4. **日志异常**
   - 只看：错误前后若干行、完整 stacktrace、Caused by 链
   - 不看：全窗日志

## 数据流示例

### 场景一：进程异常

```
1. 采集层
   ↓ 采集 process-3381 的 CPU、内存、重启次数

2. 感知层
   ↓ 检测到异常：重启 4 次、内存上升 500MB
   ↓ 输出概况包：
   {
     "entity_id": "process-3381",
     "severity": "HIGH",
     "summary": "疑似 OOM，建议查进程详情和日志",
     "suggested_tools": ["get_process_detail", "get_logs"]
   }

3. 上下文工程层
   ↓ 排序、压缩、注入 LLM

4. LLM 层
   ↓ 读到概况包，知道"process-3381 异常"
   ↓ 调用 get_process_detail(process-3381, mode=summary)
   ↓ 得到关键证据：CPU 85%、内存 2.3GB、重启 4 次
   ↓ 信息不足，调用 get_logs(log-group-77, mode=detail)
   ↓ 得到 OOM 错误日志
   ↓ 输出诊断结论
```

### 场景二：磁盘异常

```
感知层输出：
{
  "entity_id": "host-12:/var",
  "severity": "CRITICAL",
  "summary": "/var 分区使用率 93%，过去 5 分钟写入速率异常升高",
  "suggested_tools": ["get_disk_detail", "get_process_detail"]
}

LLM 调用流程：
1. get_disk_detail(host-12:/var, mode=summary)
   → 得到：使用率 93%、Top 写入进程是 process-9911
2. get_process_detail(process-9911, mode=summary)
   → 得到：该进程在疯狂写日志
3. 输出建议：清理日志或扩容
```

## 工具查询模式

### 两种查询方式

1. **按实体查**（单个对象）
   ```
   get_process_detail(pid=3381, mode=summary)
   get_disk_detail(mount_id="host-12:/var", mode=detail)
   ```

2. **按类别查**（对象集合）
   ```
   list_abnormal_processes(host_id="host-12")
   list_high_disk_usage(threshold=80%)
   list_high_packet_loss_interfaces()
   ```

## 实现建议

### 优先级

1. **第一阶段**：实现感知层 + 基础工具（summary 模式）
2. **第二阶段**：实现 detail 模式 + 上下文工程层
3. **第三阶段**：实现 full 模式 + 按类别查询

### 关键点

- ✅ 统一 JSON 协议，不然工具输出会不稳定
- ✅ 感知层必须带 `confidence` 和 `suggested_mode`，不能完全依赖 LLM 自己判断
- ✅ `full` 模式不能太自由，需要权限控制和时间窗口限制
- ✅ 每个工具都要支持 `time_window` 参数，便于时间范围查询

---

## 方案优点

| 优点 | 说明 |
|------|------|
| **工程化强** | 接口清晰，容易实现和测试 |
| **适合运维** | 运维天然就是看对象，不是看大段原始文本 |
| **省上下文** | 默认摘要，按需展开，降低 token 消耗 |
| **易权限控制** | 不同模式可对应不同权限等级 |
| **可追溯** | 每个异常都有证据指针，便于审计 |

---

## 方案风险点

| 风险 | 缓解方案 |
|------|---------|
| full 模式 token 爆炸 | 限制时间窗口、对象数量、需显式确认 |
| LLM 误判升档时机 | 感知层带 confidence 和 suggested_mode |
| 工具输出不一致 | 统一 JSON 协议，严格 schema 验证 |
| 感知层误检 | 定期校准基线，引入人工反馈 |

---

## 一句话总结

**感知层输出对象级概况包，工具层提供对象级分层详情，上下文工程层控制注入质量，LLM 负责推理和按需深入。**
