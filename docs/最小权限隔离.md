最小权限原则是指用户应拥有完成任务所需的最小访问权限，无论访问权限来自人还是系统。在开发生命周期的初期，也就是新功能的设计阶段，添加这些限制是最有效的。不必要的权限会导致可能出现的错误、漏洞或破坏的表面区域不断扩大，并造成安全和可靠性风险，而在运行系统中要控制或最小化这些风险是非常昂贵的。

在本章中，我们将讨论如何根据风险对访问权限进行分类，并研究执行最小权限的最佳实践。一个配置分配示例展示了这些实践在实际环境中的权衡。在详细介绍了身份验证和授权的策略框架后，我们深入探讨了高级授权控制，这些控制可以降低账户被入侵的风险和值班工程师犯错的风险。最后，我们承认在设计最低权限时可能会出现权衡和矛盾。在理想的世界里，使用系统的人都是用心良苦的，他们以完全安全的方式执行工作任务，不会出错。遗憾的是，现实情况却大相径庭。工程师可能会犯错，他们的账户可能会被泄露，或者他们可能是恶意的。因此，在设计系统时必须考虑到最小权限：系统应限制用户访问完成手头任务所需的数据和服务。

公司往往希望假定他们的工程师是出于好意，并依靠他们完美地执行艰巨的任务。这种期望并不合理。作为一个思考练习，想一想如果你想做坏事，会给你的组织造成多大的损失。你能做什么？怎么做？你会被发现吗？你能掩盖你的行踪吗？或者，即使你的意图是好的，你（或拥有同等权限的人）可能犯的最严重的错误是什么？在调试、应对故障或执行应急响应时，您或您的同事使用的临时手动命令中，有多少会因为一个错字或复制粘贴失败而导致或加剧故障？

因为我们不能依赖人类的完美，所以我们必须假设任何可能的不良行为或结果都有可能发生。因此，我们建议在设计系统时尽量减少或消除这些不良行为的影响。

即使我们普遍信任访问我们系统的人，我们也需要限制他们的权限和我们对他们凭证的信任。事情可能出错，也一定会出错。人们会犯错，会用胖手指发出命令，会受到攻击，也会上当受骗。期望完美是不切实际的假设。换句话说，引用 SRE 的格言--希望不是策略。

概念和术语
在深入探讨设计和运行门禁系统的最佳实践之前，让我们先为业内和谷歌使用的一些专业术语确定工作定义。

最低特权
最小权限是一个广泛的概念，在安全行业中已得到广泛认可。本章中的高层次最佳实践可以为系统奠定基础，使其能够为任何给定任务或操作路径授予所需的最少权限。这一目标适用于分布式系统中的人类、自动任务和单个机器。最小权限的目标应贯穿系统的所有身份验证和授权层。特别是，我们推荐的方法拒绝将隐式权限扩展到工具（如"工作示例：配置分发 "中所示），并尽可能确保用户不拥有环境权限--例如，以 root 身份登录的能力。

零信任 Network+
我们讨论的设计原则首先是零信任网络--即用户的网络位置（在公司网络内）不授予任何特权访问。例如，插入会议室的网络端口并不比从互联网上的其他地方连接获得更多的访问权限。相反，系统会根据用户凭证和设备凭证--我们对用户的了解和对设备的了解--的组合授予访问权限。谷歌已通过其BeyondCorp 计划成功实施了大规模零信任网络模型。

零接触
谷歌的 SRE 组织正致力于通过自动化实现最小权限的概念，目标是转向我们所说的 "零接触 "界面。这些接口的具体目标--如第 3 章所述的零接触生产（ZTP）和零接触网络（ZTN）--是通过消除人类对生产角色的直接访问，使 Google 更安全并减少中断。取而代之的是，人类通过工具和自动化对生产基础架构进行可预测和可控制的更改，从而间接访问生产。这种方法需要广泛的自动化、新的安全 API 和弹性多方审批系统。

根据风险对访问进行分类
任何降低风险的策略都需要权衡利弊。降低人为因素带来的风险可能需要额外的控制或工程工作，并可能对生产率产生影响；可能会增加工程时间、流程变更、运营工作或机会成本。您可以通过明确保护范围和优先次序来帮助限制这些成本。

并非所有数据或操作都是相同的，访问的构成可能会因系统的性质而大不相同。因此，你不应该以相同的程度保护所有访问。为了应用最合适的控制措施，避免全有或全无的心态，您需要根据影响、安全风险和/或关键性对访问进行分类。例如，您可能需要对不同类型的数据（公开数据、公司数据、用户数据和加密机密）的访问进行不同的处理。同样，您可能需要区别对待可删除数据的管理 API 和特定服务的读取 API。

您的分类应定义明确、应用一致、易于理解，这样人们在设计系统和服务时才能 "使用 "这种语言。您的分类框架将根据系统的规模和复杂程度而有所不同：您可能只需要两三种依赖于临时标签的类型，也可能需要一个强大的程序化系统，用于在中央清单中对系统的部分内容（API 分组、数据类型）进行分类。这些分类可能适用于数据存储、API、服务或用户在工作过程中可能访问的其他实体。确保您的框架能够处理系统中最重要的实体。

一旦建立了分类基础，就应该考虑对每种分类的控制。您需要考虑几个方面：

谁有权访问？

应该如何严格控制访问权限？

用户需要什么类型的访问权限（读/写）？

有哪些基础设施控制措施？

可靠性与安全性的交叉点权限
从可靠性的角度来看，您可能会从基础架构控制入手，因为您要确保未经授权的行为者无法关闭作业、更改 ACL 和错误配置服务。不过，请记住，从安全角度来看，读取敏感数据通常也会造成同样的危害：过于宽泛的读取权限可能会导致大规模数据泄露。

您的目标应该是构建一个访问框架，在此基础上应用适当的控制措施，在工作效率、安全性和可靠性之间取得适当的平衡。最低权限应适用于所有数据访问和操作。从这个基础框架出发，让我们来讨论一下如何按照最小权限的原则和控制措施设计系统。

最佳做法
在实施最小权限模式时，我们推荐几种最佳做法，详见此处。

小型功能性应用程序接口
让每个程序做好一件事。要完成一项新的工作，应重新构建程序，而不是通过添加新的 "功能 "来使旧程序复杂化。

-麦克罗伊、平森和塔格（1978 年）1

正如这句话所表达的，Unix 文化的核心是可以组合使用的小型简单工具。由于现代分布式计算已从 20 世纪 70 年代的单个分时计算系统发展为覆盖整个地球的网络连接分布式系统，因此作者的建议在 40 多年后的今天依然适用。为了将这句话应用到当前的计算环境中，我们可以这样说："让每个 API 端点做好一件事"。在着眼于安全性和可靠性构建系统时，应避免开放式交互界面，而应围绕小型功能 API 进行设计。这种方法允许您应用经典的 "最小权限 "安全原则，授予执行特定功能所需的最小权限。

API 究竟是什么意思？每个系统都有一个 API：它只是系统提供的用户界面。有些 API 非常庞大（如 POSIX API2或 Windows API3），有些则相对较小（如 memcached4和 NATS5等），有些则非常小（如World Clock API、TinyURL、6和谷歌字体 API7).当我们谈论分布式系统的应用程序接口（API）时，我们指的只是可以查询或修改其内部状态的所有方式的总和。API 的设计在计算机文献中已有详尽论述；8本章将重点讨论如何通过使用少量定义明确的基元来暴露 API 端点，从而设计并安全地维护安全系统。例如，您评估的输入可能是对唯一 ID 的 CRUD（创建、读取、更新和删除）操作，而不是接受编程语言的 API。

除了面向用户的 API 之外，还要仔细关注管理 API。管理 API 对于应用程序的可靠性和安全性同样重要（可以说更加重要）。使用这些 API 时的错字和错误可能会导致灾难性的中断或暴露大量数据。因此，管理 API 也是恶意行为者最具吸引力的攻击面。

管理应用程序接口仅由内部用户和工具访问，因此相对于面向用户的应用程序接口，它们可以更快、更容易地进行更改。不过，在内部用户和工具开始使用任何 API 后，更改 API 仍需付出一定的成本，因此我们建议仔细考虑 API 的设计。管理 API 包括以下内容：

设置/维护应用程序接口，例如用于构建、安装和更新软件或配置软件运行容器的应用程序接口

维护和应急应用程序接口，如删除损坏的用户数据或状态或重启行为不端进程的管理访问权限

当涉及访问和安全性时，API 的大小是否重要？请看一个我们熟悉的例子：POSIX API，这是我们之前举过的一个大型 API 的例子。这种 API 很受欢迎，因为它既灵活又为许多人所熟悉。作为生产机器管理 API，它最常用于相对明确的任务，如安装软件包、更改配置文件或重启守护进程。

用户通常通过交互式 OpenSSH 会话或使用针对 POSIX API 编写脚本的工具9主机的设置和维护，或者使用针对 POSIX API 编写脚本的工具。这两种方法都会将整个 POSIX API 暴露给调用者。在这种交互会话中，很难限制和审计用户的操作。如果用户恶意试图规避控制，或者连接的工作站受到威胁，情况就更是如此。

我不能只审核互动环节吗？
许多简单的审计方法，如捕获 bash 执行的命令或用script(1) 封装交互式会话，看似已经足够，而且相当全面。然而，这些用于交互会话的基本日志记录选项就像房子门上的锁：它们只能让诚实的人保持诚实。不幸的是，知道它们存在的攻击者可以轻而易举地绕过大多数这些类型的基本会话审计。

举个简单的例子，攻击者可以通过打开 vim 等编辑器并在交互会话中执行命令（如:!/bin/evilcmd ）来绕过 bash 命令历史记录。这种简单的攻击也会使你通过script(1), 检查 Typescript 日志输出的能力变得复杂，因为输出会被 vim 和 ncurses 用来显示可视化编辑环境的可视化控制字符弄得一团糟。我们可以使用更强大的机制，例如捕获内核审计框架的部分或全部系统调用，但值得了解的是天真方法的缺点，以及实现更强大方法的难度。

您可以使用各种机制来限制通过 POSIX API 授予用户的权限、10但这种必要性是公开一个非常庞大的 API 的根本缺陷。相反，最好的办法是将这个庞大的管理 API 简化和分解成更小的部分。然后，你就可以遵循最小权限原则，只为任何特定调用者所需的特定操作授予权限。

备注
暴露的 POSIX API 不应与 OpenSSH API 相混淆。在不公开整个 POSIX API 的情况下，也可以利用 OpenSSH 协议及其身份验证、授权和审计（AAA）控制；例如，使用Git-shell。

碎玻璃
碎玻璃装置的名称来源于指示用户 "在紧急情况下打碎玻璃 "的火警拉杆，它可以在紧急情况下访问系统，并完全绕过授权系统。这对于从意外情况中恢复非常有用。有关更多信息，请参阅"优雅故障和玻璃窗机制 "和"诊断拒绝访问"。

审计
审计主要用于检测不正确的授权使用。这可能包括恶意系统操作员滥用权力、外部人员泄露用户凭证，或流氓软件对另一个系统采取意想不到的行动。审计和有意义地检测噪声中信号的能力在很大程度上取决于所审计系统的设计：

做出或绕过访问控制决定的粒度如何？(什么？）

如何清楚地获取与请求相关的元数据？谁？

以下提示有助于制定合理的审计策略。最终，您的成功还将取决于与审计相关的文化。

收集良好的审计日志
使用小型功能应用程序接口（如"小型功能应用程序接口 "中所述）可为审计能力提供最大的优势。最有用的审计日志会捕获一系列细粒度的操作，例如 "推送加密哈希值为 123DEAD...BEEF456 的配置 "或 "执行<x> 命令"。考虑如何向客户展示和说明您的管理操作，也有助于使审计日志更具描述性，从而在内部发挥更大作用。细粒度的审计日志信息可以有力地证明用户进行了或没有进行哪些操作，但一定要专注于捕获操作中有用的部分。

特殊情况需要特殊的访问权限，这就需要有强大的审计文化。如果您发现现有的小型功能 API 表面不足以恢复系统，您有两个选择：

提供断线功能，允许用户打开交互会话，使用强大而灵活的应用程序接口。

允许用户直接访问凭证，从而排除对凭证使用情况的合理审计。

在这两种情况下，您都可能无法建立详细的审计跟踪。记录用户打开了大型 API 的交互会话，并不能有效地告诉你他们做了什么。有动机且知识渊博的内部人员可以轻易绕过许多捕获交互会话会话日志的解决方案，例如记录 bash 命令历史记录。即使可以捕获完整的会话记录，有效地对其进行审计也可能相当困难：使用 ncurses 的可视化应用程序需要重放才能让人看懂，而 SSH 多路复用等功能会使捕获和理解交错状态变得更加复杂。

对于过于宽泛的应用程序接口和/或频繁使用的 "突破镜"，解药是培养一种重视仔细审核的文化。这对于可靠性和安全性都至关重要，你可以利用这两个动机来吸引责任方。两双眼睛有助于避免错别字和错误，而且应始终防止单方面访问用户数据。

最终，构建管理应用程序接口和自动化的团队需要以有利于审计的方式设计这些应用程序接口和自动化。任何经常访问生产系统的人都应受到激励，以共同解决这些问题，并了解良好审计日志的价值。如果不加强文化建设，审计就会成为橡皮图章，而使用防爆玻璃也会成为家常便饭，失去其重要性和紧迫性。文化是确保团队以支持审计的方式选择、构建和使用系统的关键；是确保这些事件很少发生的关键；也是确保审计事件受到应有审查的关键。

选择审计师
收集好审计日志后，您需要选择合适的人员来检查记录的事件（希望很少发生）。审计人员需要有正确的背景和目标。

说到上下文，审计员需要知道特定操作的作用，最好还要知道操作者为什么需要执行该操作。因此，审核员通常是团队成员、经理或熟悉需要执行该操作的工作流程的人。您需要在充分的背景和客观性之间取得平衡：内部审核员可能与产生审核事件的人有密切的私人关系，并且/或者希望组织取得成功，而外部私人审核员可能希望继续被组织聘用。

选择目标正确的审计师取决于审计的目的。在谷歌，我们执行两大类审计：

审计以确保最佳做法得到遵循

审计以查明安全漏洞

一般来说，"最佳实践 "审计支持我们的可靠性目标。例如，一个 SRE 团队可能会选择在每周的团队会议上审核上周值班的 breakglass 事件。这种做法提供了一种文化上的同行压力，促使我们使用并改进较小的服务管理 API，而不是使用断裂镜机制来访问更灵活的应急使用 API。大范围的 "玻璃窗 "访问通常会绕过部分或全部安全检查，从而使服务面临更高的人为错误风险。

谷歌通常会将 "破镜重圆 "审查下放到团队层面，我们可以利用团队审查所带来的社会规范。执行审查的同级人员具有一定的背景知识，即使是伪装得很好的行为，他们也能发现，这对于防止内部滥用和挫败恶意内部人员至关重要。例如，如果某个同事反复使用 "破镜重圆 "操作来访问他们实际上可能并不需要的异常资源，同行就能很好地发现。这种团队审查还有助于识别管理 API 中的缺陷。当特定任务需要使用断桥访问时，往往意味着需要提供一种更安全或更可靠的方式来执行该任务，作为正常 API 的一部分。您可以在第 21 章中阅读更多相关内容。

在谷歌，我们倾向于集中进行第二种类型的审计，因为识别外部安全漏洞可以从组织的广阔视野中获益。高级攻击者可能会入侵一个团队，然后利用该访问权限入侵另一个团队、服务或角色。每个单独的团队可能不会注意到一些异常行为，也没有跨团队视图来连接不同行为之间的点。

中央审计团队还可以为不广为人知的其他审计事件建立额外的信号和添加代码。这些类型的绊网在早期检测中特别有用，但您可能不想广泛分享其实施细节。您可能还需要与组织内的其他部门（如法律和人力资源部门）合作，以确保审计机制的适当性、范围的正确性和文档的完整性。

在谷歌，我们使用结构化理由将结构化数据与审计日志事件相关联。当产生审计日志的事件发生时，我们可以将其与结构化参考相关联，例如错误编号、票据编号或客户案例编号。这样，我们就可以对审计日志进行程序检查。例如，如果支持人员查看客户的付款详情或其他敏感数据，他们可以将这些数据与特定的客户案例关联起来。因此，我们可以确保观察到的数据属于打开案例的客户。如果我们依赖自由文本字段，日志验证的自动化就会难得多。结构化说明是我们扩展审计工作的关键--它提供了对有效审计和分析至关重要的集中式审计团队上下文。

或者，假设您正在开发一款键盘应用程序，允许用户一键发布备忘录。您希望分析用户的行为和历史记录，以便自动推荐备忘录。由于缺乏适当的测试基础架构，您需要让数据分析师读/写访问生产中的一整套原始用户数据，以便执行分析和测试。

诊断拒绝访问
在复杂的系统中，执行的是最低权限，客户必须通过第三因素、多方授权或其他机制（见"高级控制"）来赢得信任，因此策略的执行是多层次和细粒度的。因此，策略拒绝也可能以复杂的方式发生。

考虑这样一种情况：正在执行合理的安全策略，而授权系统却拒绝访问。可能会出现以下三种结果之一

客户已被正确拒绝，您的系统也采取了适当措施。最低权限已得到执行，一切正常。

客户端被正确拒绝，但可以使用高级控制（如多方授权）来获得临时访问权。

客户认为他们被错误地拒绝了，并可能向安全策略团队提交支持票据。例如，如果客户最近被从授权组中删除，或者如果策略发生了微妙或可能不正确的变化，就可能发生这种情况。

在所有情况下，来电者都对拒绝的原因一无所知。但系统是否可以向客户提供更多信息呢？根据呼叫者的权限级别，可以这样做。

如果客户端没有权限或权限非常有限，则拒绝应保持盲目--您可能不想暴露 403 Access Denied 错误代码（或等同代码）以外的信息，因为有关拒绝原因的详细信息可能会被利用来获取系统信息，甚至找到获取访问权限的方法。不过，如果调用者拥有某些最低权限，则可以提供与拒绝相关的令牌。调用者可以使用该令牌调用高级控制以获得临时访问权限，或者通过支持渠道将令牌提供给安全策略团队，以便他们使用该令牌诊断问题。对于权限较高的呼叫者，可以提供与拒绝相关联的令牌和一些补救信息。然后，呼叫者可以在调用支持通道之前尝试自我修复。例如，来电者可能会了解到，访问被拒绝是因为他们需要成为某个特定组的成员，然后他们就可以请求访问该组。

暴露多少修复信息与安全策略团队能够承受多少超负荷支持之间总是存在矛盾。但是，如果暴露的信息过多，客户可能会根据拒绝信息重新设计策略，从而使恶意行为者更容易以非预期的方式提出使用策略的请求。有鉴于此，我们建议在实施零信任模式的早期阶段，使用令牌并让所有客户端调用支持通道。

优雅故障和玻璃破碎机制
在理想情况下，你总是在使用一个有效的授权系统来执行合理的策略。但在现实中，你可能会遇到导致大规模错误拒绝访问的情况（可能是由于系统更新不及时）。对此，您需要通过断桥机制来规避授权系统，以便对其进行修复。

在使用防爆玻璃装置时，应考虑以下指导原则：

使用断网机制的能力应受到严格限制。一般来说，只有负责系统运行 SLA 的 SRE 团队才能使用。

零信任网络的防玻璃机制只能在特定地点使用。这些地点就是你的应急室，是具有额外物理访问控制的特定地点，以抵消对其连接性增加的信任。(细心的读者会注意到，零信任联网的后备机制，即不信任网络位置的策略，是......信任网络位置--但有额外的物理访问控制）。

应密切监控防碎玻璃装置的所有使用情况。

负责生产服务的团队应定期测试玻璃破碎装置，以确保在需要时能正常工作。

当成功利用 "断镜 "机制使用户重新获得访问权限后，您的 SRE 和安全策略团队就可以进一步诊断和解决根本问题。第8章和第 9章将讨论相关策略。

通过 OpenSSH 的 POSIX 应用程序接口
您可以允许自动化程序通过 OpenSSH 连接到网络服务器主机，通常以网络服务器运行的本地用户身份进行连接。然后，自动化程序就可以写入配置文件并重启网络服务器进程。这种模式简单而常见。它利用的管理 API 很可能已经存在，因此几乎不需要额外的代码。遗憾的是，利用现有的大型管理 API 会带来一些风险：

运行自动化的角色可以永久停止网络服务器，启动另一个二进制文件取而代之，读取它可以访问的任何数据等。

自动化系统中的漏洞隐含着足够的访问权限，可以导致所有网络服务器协调中断。

自动化凭据的泄露等同于所有网络服务器的泄露。

您可以使用与更新网络服务器二进制文件相同的机制，将配置作为打包的软件更新发布。打包和触发二进制更新的方法有很多，使用的 API 也各不相同。一个简单的例子是，通过从cron 调用周期性的apt-get ，从中央存储库中提取 Debian 软件包（.deb）。您可以使用以下章节中讨论的模式之一构建一个更复杂的示例来触发更新（而不是cron ），然后将其重复用于配置和二进制文件。随着二进制文件分发机制的发展，安全性和保障性也会随之增加，因为两者使用的是相同的基础架构。为集中协调金丝雀进程、协调健康检查或提供签名/验证/审计所做的任何工作，同样也会为这两种工件带来收益。

有时，二进制文件和配置更新系统的需求并不一致。例如，您可能会在配置中分发一个 IP 拒绝列表，该列表需要在安全可行的情况下尽快收敛，同时还要将网络服务器二进制文件构建到一个容器中。在这种情况下，以与分发配置更新相同的速度构建、安装和拆卸一个新容器，可能成本太高或会造成干扰。这种类型的需求冲突可能需要两种分发机制：一种用于二进制文件，另一种用于配置更新。

权衡利弊
除了表 5-2中的大型选项外，其他选项都能为自动化提供安全保障。攻击者仍有可能通过推送任意配置来破坏网络服务器的角色；不过，选择较小的 API 意味着推送机制不会隐含地允许这种破坏。

通过对配置进行独立于推送配置的自动化签名，可以进一步实现最小权限设计。这种策略可以分割角色之间的信任关系，确保如果推送配置的自动化角色受到威胁，自动化角色也无法通过发送恶意配置来威胁网络服务器。回顾 McIlroy、Pinson 和 Tague 的建议，设计系统的每一部分来执行一项任务，并出色地完成这项任务，这样就能隔离信任。

使用高级授权控制
给定资源的访问控制列表是实现授权决策的常用方法。最简单的 ACL 是一个与验证角色相匹配的字符串，通常与一些分组概念相结合--例如，一组角色（如 "管理员"），然后扩展为一个更大的角色列表（如用户名）。当服务评估传入的请求时，它会检查经过验证的角色是否是 ACL 的成员。

更复杂的授权要求，如多因子授权（MFA）或多方授权（MPA），需要更复杂的授权代码（有关三因素授权和 MPA 的更多信息，请参阅"高级控制"）。此外，一些组织在设计授权策略时可能需要考虑其特定的监管或合同要求。

这些代码可能很难正确实施，如果许多服务各自实施自己的授权逻辑，其复杂性就会迅速增加。根据我们的经验，使用AWS或GCP身份与访问管理（IAM）等框架，将复杂的授权决策与核心 API 设计和业务逻辑分离开来是很有帮助的。在谷歌，我们也在内部服务中广泛使用 GCP 授权框架的变体。15

安全策略框架允许我们的代码进行简单的检查（如 "X 能否访问资源 Y？"），并根据外部提供的策略对这些检查进行评估。如果我们需要在特定操作中添加更多授权控制，只需更改相关的策略配置文件即可。这种低开销带来了巨大的功能和速度优势。

投资于广泛使用的授权框架
通过使用共享库来执行授权决策，并尽可能广泛地使用一致的接口，可以大规模地实现身份验证和授权变更。将这一经典的模块化软件设计建议应用到安全领域，会产生令人惊喜的效果。例如

只需更改一个库，就能为所有服务端点添加对 MFA 或 MPA 的支持。

然后，您只需更改一次配置，就能为所有服务中的一小部分操作或资源提供这种支持。

您可以通过要求对所有允许潜在不安全操作的操作进行 MPA（类似于代码审查系统）来提高可靠性。这种流程改进可以在不允许广泛的单方面访问的情况下，通过促进快速事件响应（通过绕过修订控制系统和代码审查依赖关系），提高抵御内部风险威胁的安全性（有关对手类型的更多信息，请参阅第 2 章）。

随着企业的发展，标准化是您的朋友。统一的授权框架有利于团队的流动性，因为更多的人知道如何根据通用框架编码和实施访问控制。

避免潜在陷阱
设计复杂的授权策略语言非常困难。如果策略语言过于简单，就无法实现目标，最终会导致授权决策分散在框架策略和主代码库中。如果策略语言过于笼统，就很难进行推理。为了减少这些问题，您可以采用标准的软件 API 设计实践，尤其是迭代设计方法，但我们建议您谨慎行事，避免出现上述两种极端情况。

仔细考虑如何将授权策略与二进制文件一起发送。您可能希望独立于二进制文件更新授权策略，因为授权策略可能会成为对安全最敏感的配置之一。有关配置分发的其他讨论，请参阅上一节中的工作示例、本书第 9章和第 14章、SRE 书籍第 8 章以及 SRE 工作手册第14章和第15章。

应用程序开发人员将需要帮助来处理这种语言所编码的策略决策。即使您避免了此处所述的陷阱，创建了一种表达力强且易于理解的策略语言，但通常仍需要实施管理 API 的应用程序开发人员与掌握生产环境特定领域知识的安全工程师和 SRE 之间进行协作，才能在安全性和功能性之间取得适当的平衡。

深潜：高级控制
虽然许多授权决定都是 "是/否 "二选一，但在某些情况下，更多的灵活性是有用的。与其严格要求 "是/否"，不如设置一个 "也许 "的逃生阀，再配以额外的检查，这样可以大大减轻系统的压力。这里介绍的许多控制措施既可以单独使用，也可以组合使用。适当的使用取决于数据的敏感性、操作的风险和现有的业务流程。

多方授权（MPA）
让另一个人参与进来是确保做出正确访问决定、培养安全可靠文化的一个经典方法（见第 21 章）。这种策略有几个好处：

防止可能导致安全或隐私问题的错误或无意违反政策的行为。

阻止坏人试图进行恶意更改。这既包括面临纪律处分风险的员工，也包括面临被发现风险的外部攻击者。

增加攻击的成本，要求至少有一个人受到攻击，或者是精心设计的、通过同行审查的改动。

对过去的行动进行审计，以进行事件响应或事后分析，前提是以防篡改的方式对审查进行永久记录。

让客户放心。如果您的客户知道没有一个人可以独自做出改变，他们可能会更乐意使用您的服务。

MPA 通常针对广泛的访问级别执行--例如，要求获得批准才能加入允许访问生产资源的组，或者要求获得作为特定角色或凭证行事的能力。宽泛的 MPA 可以作为一种有价值的 "玻璃窗 "机制，用于执行可能没有特定工作流程的非常规操作。在可能的情况下，应尽量提供更细粒度的授权，这样可以提供更强的可靠性和安全性保证。如果第二方批准了针对小型功能 API 的操作（请参阅"小型功能 API"），那么他们就可以更有信心地确定他们所授权的内容。

潜在的陷阱
确保审批人有足够的背景信息来做出明智的决定。所提供的信息是否能明确指出谁在做什么？您可能需要指定配置参数和目标，以显示命令在做什么。如果您在移动设备上显示审批提示，而屏幕空间有限，无法显示详细信息，那么这一点可能尤为重要。

审批方面的社会压力也可能导致错误的决策。例如，如果一个可疑请求是由经理、高级工程师或站在他们办公桌前的人发出的，工程师可能会觉得不便拒绝。为了减轻这些压力，您可以提供在事后将审批升级到安全或调查团队的选项。或者，您可以制定一项政策，规定对所有（或一定比例）某类审批进行独立审核。

在建立多方授权系统之前，要确保技术和社会动态允许有人说 "不"。否则，系统就没有什么价值。

三因素授权（3FA）
在大型组织中，MPA 通常有一个关键弱点，会被意志坚定且持久的攻击者利用：所有 "多方 "都使用相同的集中管理工作站。工作站的同质性越高，入侵一个工作站的攻击者就越有可能入侵多个甚至所有工作站。

加固工作站机群以防攻击的经典方法是让用户维护两台完全独立的工作站：一台用于一般用途，如浏览网页和收发电子邮件，另一台更可信的工作站用于与生产环境通信。根据我们的经验，用户最终希望从这些工作站获得类似的功能和能力，而为有限的用户维护两套工作站基础设施，其证书需要更高水平的保护，既昂贵又难以长期维持。一旦这个问题不再是管理层最关心的问题，人们维护基础架构的积极性就会降低。

要降低单个受损平台破坏所有授权的风险，需要采取以下措施：

维护至少两个平台

在两个平台上批准申请的能力

(最好）能够硬化至少一个平台

考虑到这些要求，另一种方法是要求对某些非常危险的操作从加固的移动平台进行授权。为简单方便起见，，你可以只允许 RPC 源自完全托管的桌面工作站，然后要求移动平台提供三因素授权。当生产服务接收到敏感的 RPC 时，策略框架（在"身份验证和授权决策的策略框架 "中进行了描述）会要求单独的 3FA 服务提供加密签名批准。然后，该服务会指出，它向移动设备发送了 RPC，并向发起用户展示了该 RPC，用户也确认了该请求。

加固移动平台比加固通用工作站要容易一些。我们发现，用户通常对移动设备上的某些安全限制更宽容，比如额外的网络监控、只允许一组应用程序、通过有限的 HTTP 端点进行连接等。这些策略在现代移动平台上也很容易实现。

一旦你有了可以显示生产变更建议的加固移动平台，你就必须将请求发送到该平台，并将其显示给用户。在谷歌，我们重复使用向安卓手机发送通知的基础架构，以便向用户授权和报告谷歌登录尝试。如果你身边有类似的加固基础架构，将其扩展以支持此用例可能会很有用，但如果没有，创建一个基于网络的基本解决方案也相对容易。3FA 系统的核心是一个简单的 RPC 服务，它接收授权请求，并将请求提供给受信任的客户端进行授权。请求使用受 3FA 保护的 RPC 的用户会从其移动设备访问 3FA 服务的 Web URL，并收到批准请求。

重要的是要区分 MPA 和 3FA 可防范哪些威胁，这样才能就何时应用它们制定一致的政策。MPA 可防范单方面的内部风险，也可防范单个工作站被入侵（需要第二次内部批准）。3FA 可以防止内部工作站被广泛入侵，但单独使用时无法防止内部威胁。要求发起者提供 3FA 并要求第二方提供简单的基于网络的 MPA，可以提供非常强大的防御能力，抵御大多数这些威胁的组合，而且组织开销相对较小。

3FA 不是 2FA（或 MFA）
双因素身份验证（2FA）是多因素身份验证的一个广受讨论的子集。具体来说，它试图将 "你知道的东西"（密码）与 "你拥有的东西"（能产生密码存在证明的应用程序或硬件令牌）结合起来，形成一个强大的身份验证决策。有关 2FA 的更多信息，请参阅案例研究"示例：使用 FIDO 安全密钥的强第二因素身份验证"。

主要区别在于，3FA 试图对特定请求提供更强的授权，而不是对特定用户提供更强的身份验证。虽然我们承认3FA中的3有点名不副实（它是来自第二个平台的批准，而不是第三个平台），但 "3FA 设备 "是指在前两个因素之外为某些请求增加额外授权的移动设备，这是一个有用的简写。

业务理由
正如在"选择审核员 "中提到的，您可以通过将访问与结构化的业务理由（如错误、事件、票据、案例 ID 或指定账户）绑定来执行授权。但建立验证逻辑可能需要额外的工作，还可能需要对值班人员或客户服务人员的流程进行更改。

以客户服务工作流程为例。在小型或不成熟的组织中，有时会发现一种反模式，即出于效率原因或由于不存在控制措施，基本的新生系统可能会允许客户服务代表访问所有客户记录。更好的办法是默认阻止访问，只有在能够验证业务需求时才允许访问特定数据。随着时间的推移，这种方法可能会逐步实施控制。例如，一开始可以只允许分配给未结票单的客户服务代表访问。随着时间的推移，您可以改进系统，只允许访问特定客户和这些客户的特定数据，并以有时间限制的方式，经客户批准。

如果配置得当，该策略可提供强大的授权保证，确保访问是适当的，范围是正确的。结构化理由允许自动化要求 #12345 号票据不是为了满足简单的正则表达式检查而输入的随机数。取而代之的是，该理由要满足一套访问策略，在操作业务需求和系统功能之间取得平衡。

安全性与可靠性的权衡：系统可用性
如果工程师每次要执行权限提升的操作时都很难找到批准人，他们就会产生潜在的危险行为。例如，他们可能会通过提供通用的业务理由（如 "Team foo 需要访问权限"）来规避强制执行的最佳实践，而这并不符合功能要求。通用理由模式应触发审计系统中的报警信号。

临时通道
您可以通过授予资源临时访问权来限制授权决策的风险。当无法对每项操作进行精细控制，但仍想利用现有工具授予尽可能少的权限时，这种策略往往很有用。

您可以以结构化和计划的方式（如在值班轮换期间，或通过到期的群组成员身份），或以用户明确提出访问请求的按需方式，授予临时访问权限。你可以将临时访问与多方授权请求、业务理由或其他授权控制结合起来。临时访问也为审计提供了一个逻辑点，因为你可以清楚记录在任何特定时间拥有访问权限的用户。它还提供了有关临时访问发生位置的数据，这样你就可以确定优先级，并随着时间的推移减少这些请求。

临时访问也会降低环境权限。这也是管理员青睐sudo 或 "以管理员身份运行 "而不是以 Unix 用户root或 Windows 管理员账户进行操作的原因之一--当你不小心发出删除所有数据的命令时，权限越少越好！

代理
当无法对后端服务进行细粒度控制时，可以使用受到严格监控和限制的代理服务器（或堡垒）。只有来自这些指定代理的请求才允许访问敏感服务。这种代理可以限制危险操作、限制操作速率，并执行更高级的日志记录。

例如，您可能需要对错误变更执行紧急回滚。鉴于发生错误变更的方式和解决方式不一而足，执行回滚所需的步骤可能无法在预定义的 API 或工具中找到。你可以给予系统管理员解决紧急情况的灵活性，但要引入限制或额外的控制措施来降低风险。例如

每条命令都可能需要同行批准。

管理员只能连接相关机器。

管理员使用的计算机可能无法访问互联网。

您可以启用更全面的日志记录。

正如下一节所讨论的那样，实施任何这些控制措施都会产生集成和运行成本。

权衡与矛盾
采用最小权限访问模式肯定会改善组织的安全状况。但是，您必须将前几节所述的好处与实施该模式的潜在成本相抵消。本节将讨论其中的一些成本。

安全复杂性增加
高度细化的安全态势是一个非常强大的工具，但它也很复杂，因此管理起来很有挑战性。拥有一套全面的工具和基础设施来帮助您定义、管理、分析、推送和调试安全策略非常重要。否则，这种复杂性可能会让人难以承受。您应该始终以能够回答这些基本问题为目标："特定用户是否有访问特定服务/数据的权限？"和 "对于特定服务/数据，谁有访问权限？"

对协作和公司文化的影响
虽然严格的最小特权模式可能适用于敏感数据和服务，但在其他领域采用更宽松的方法也能带来切实的好处。

例如，为软件工程师提供广泛的源代码访问权限会带来一定的风险。不过，工程师可以根据自己的好奇心在工作中学习，并在正常工作之外贡献自己的注意力和专业知识，从而抵消风险。不那么明显的是，这种透明度也让工程师更难写出不合适的代码而不被注意到。

将源代码和相关工件纳入数据分类工作，可以帮助您形成保护敏感资产的原则性方法，同时受益于对不太敏感资产的可见性，您可以在第 21 章中了解更多相关信息。

影响安全的高质量数据和系统
零信任环境是最小特权的基础，在零信任环境中，每一个细粒度的安全决策都取决于两点：正在执行的策略和请求的上下文。上下文由大量数据组成，其中一些数据可能是动态的，会对决策产生影响。例如，数据可能包括用户的角色、用户所属的组、发出请求的客户端属性、输入机器学习模型的训练集或所访问 API 的敏感性。您应该检查生成这些数据的系统，确保影响安全的数据质量尽可能高。低质量数据将导致错误的安全决策。

对用户生产力的影响
您的用户需要尽可能高效地完成工作流程。最好的安全态势是终端用户不会注意到的安全态势。但是，引入新的三因素和多方授权步骤可能会影响用户的工作效率，尤其是如果用户必须等待才能获得授权。您可以通过确保新步骤易于浏览来尽量减少用户的痛苦。同样，最终用户需要一种简单的方法来了解拒绝访问的原因，可以通过自助服务诊断或快速访问支持渠道。

对开发人员复杂性的影响
随着最小权限模型在整个组织内的采用，开发人员也必须与之保持一致。这些概念和策略必须让不太精通安全的开发人员也能轻松理解，因此您应该提供培训材料，并彻底记录您的应用程序接口。16在开发人员掌握新要求的过程中，让他们能够方便快捷地联系安全工程师，进行安全审查和一般咨询。在这种环境下部署第三方软件需要特别小心，因为你可能需要将软件包裹在一个可以执行安全策略的层中。

结论
在设计复杂系统时，最小权限模型是最安全的方法，可确保客户有能力完成他们需要做的事情，但不能做更多。这是一种强大的设计模式，可保护系统和数据免受已知或未知用户的恶意或意外破坏。谷歌花费了大量的时间和精力来实施这一模式。以下是其关键组成部分：

全面了解系统的功能，以便根据每个部分的安全风险程度对其进行分类。

在此基础上，对系统进行分区，尽可能精细地访问数据。小型功能应用程序接口是实现最低权限的必要条件。

验证系统，用于在用户尝试访问系统时验证他们的凭证。

授权系统可执行定义明确的安全策略，并可轻松附加到精细分区的系统中。

一套用于细微授权的高级控制。例如，可提供临时、多因素和多方审批。

系统支持这些关键概念的一系列操作要求。系统至少需要具备以下条件：

能够审计所有访问并生成信号，以便识别威胁并进行历史取证分析

推理、定义、测试和调试安全策略的方法，以及为该策略提供最终用户支持的方法

当系统出现与预期不符的情况时，能够提供破玻机制

要使所有这些组件的工作方式易于用户和开发人员采用，并且不会严重影响他们的工作效率，还需要组织承诺尽可能无缝地采用最小特权。这种承诺包括设立一个重点突出的安全职能部门，负责安全态势，并通过安全咨询、政策定义、威胁检测和安全相关问题的支持，与用户和开发人员进行沟通。
