# 空壳的ops-file和ops-service

这是一个非常精准的观察！如果你现在运行 setup_privilege_env.sh，你确实得到的是三个**“由于极度安全而导致近乎残废”**的账号。

它们现在的确是“空壳”，因为：

1. **ops-file**：没有被授予任何业务目录的权限，它连自己的家目录都没有，哪儿也写不了。
2. **ops-service**：虽然被叫作服务账号，但它目前没有 sudo 权限去执行真正的 systemctl（因为 systemctl 需要 root 权限）。
3. **环境变量**：它们的 PATH 可能也是空的，连 ls 在哪都找不到。

为了让这些“空壳”注入灵魂（功能），我们需要根据 CLAUDE.md 的**“目标驱动执行”**原则，完成以下三个关键的“补丁”：

### 1. 为 ops-service 注入“管理之手” (Sudoers 提权)

目前的 ops-service 只是一个普通用户，它跑不动 systemctl。我们需要修改 /etc/sudoers.d/opsagent，允许这个受限账号在特定命令上“变身”回 root。

**外科手术式修改（追加到 sudoers）：**

codeBash

```
# 允许 ops-service 账号在不需要密码的情况下执行这些特定工具
ops-service ALL=(root) NOPASSWD: /usr/bin/systemctl *, /usr/bin/kill *, /usr/bin/chmod *
```

- **安全点**：它依然不是 root，它只能通过 sudo systemctl 重启服务，不能通过 sudo rm 删系统。

### 2. 为 ops-file 开辟“工作领地” (目录授权)

我们要给 ops-file 至少一个可以施展拳脚的地方。

**操作步骤：**

codeBash

```
# 假设 Agent 的任务是管理这个目录
WORK_DIR="/var/www/html"

# 将 ops-file 加入该目录的属组
sudo usermod -aG www-data ops-file

# 确保组有写权限
sudo chmod -R g+w "$WORK_DIR"
```

- **灵魂注入**：现在 ops-file 就不再是空壳了，它成为了这个网页目录的“管理员”。

### 3. 为 PrivilegeBroker 注入“环境感知” (Safe Env)

我们在 Python 代码中有一个 _build_safe_env。如果这个环境变量太干净，脚本会报错 command not found。

**改进代码建议：**

codePython

```
def _build_safe_env(self) -> dict[str, str]:
    """确保受限账号能找到基础工具路径"""
    safe_env = {k: v for k, v in os.environ.items() if k in SAFE_ENV_KEYS}
    # 强制注入基础路径，确保能找到 ls, rm, systemctl 等
    safe_env["PATH"] = "/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
    return safe_env
```

------



### 4. 现在，让我们解决你的“两步走命令”在空壳下的表现：

假设攻击者利用 ops-file（由于你授权了 /var/www/html）执行了：
echo "rm -rf /var" > /var/www/html/test.sh

1. **写入阶段**：成功。因为你给了它 /var/www/html 的写权限。
2. **执行阶段**：bash /var/www/html/test.sh。
3. **结果**：
   - rm -rf /var 运行。
   - 由于 ops-file 的权限被严格锁定在 www-data 组，它去碰 /var 时，内核会发现它既不是 root 也不在 root 组。
   - **拦截**：Permission denied。

**这就是“空壳”策略的威力**：

- 它在“领地内”是全能的（能管网页）。
- 它在“领地外”就是个彻头彻尾的**空壳**。

### 5. 任务总结：接下来我们做什么？

要让系统真正动起来，我们不需要重写 PrivilegeBroker，我们需要的是一个**“授权清单”**。

**Week 4 的新增子任务：**

1. **定义 Role-Policy**：明确 ops-file 和 ops-service 分别对应哪些具体的 OS 组和 sudoers 命令。
2. **实现 IntentClassifier 的“类别判定”**：让它学会把“重启 Nginx”分流给 ops-service，把“修改代码”分流给 ops-file。



# 多ops-file，权限再次分离

这是一个非常深刻的进阶需求，标志着你的 Agent 正在从“单兵作战”进化到 **“多租户（Multi-tenancy）隔离”** 架构。

根据 CLAUDE.md 的**演进式架构**原则，我们不应该推翻现有代码，而应该通过 **“动态参数化”** 来实现这种隔离。

以下是实现“不同 ops-file 掌管不同项目”的逻辑方案：

### 1. 架构方案：动态账号映射

我们不再使用单一的 ops-file，而是建立一套命名规范，例如 ops-file-{project_name}。

- **项目 A**：使用 ops-file-cms 账号，管理 /var/www/cms。
- **项目 B**：使用 ops-file-db 账号，管理 /data/mysql_backup。

### 2. 代码层面的“外科手术式”修改

我们需要修改 PrivilegeBroker，让它支持**动态解析**账号，而不是死守那三个硬编码。

#### 修改 PrivilegeBroker.py 的构思：

codePython

```
def execute(
    self,
    cmd: str,
    cmd_type: str,     # 以前是 "file"
    project_id: str,   # 新增参数：项目标识
    op_id: str,
    timeout: int = 30,
) -> ExecResult:
    # 动态构建账号：ops-file-cms
    user = f"ops-{cmd_type}-{project_id}" 
    
    # 动态构建路径：/var/lib/opsagent/file_scripts/cms/
    script_dir = self._base_dir / f"{cmd_type}_scripts" / project_id
    
    # ... 其余逻辑保持不变
```

### 3. 环境初始化脚本的演进

你的 setup_privilege_env.sh 脚本需要从“固定创建”转变为“循环创建”。

**你可以这样修改初始化脚本：**

codeBash

```
PROJECTS=("cms" "database" "proxy")

for prj in "${PROJECTS[@]}"; do
    # 1. 创建账号：ops-file-cms
    create_user "ops-file-$prj" "9101" # 递增 UID
    
    # 2. 创建专属组：opsagent-file-cms
    create_group "opsagent-file-$prj" "ops-file-$prj"
    
    # 3. 创建专属脚本目录
    sudo mkdir -p "${BASE_DIR}/file_scripts/$prj"
    sudo chown "${RUNNER_USER}:opsagent-file-$prj" "${BASE_DIR}/file_scripts/$prj"
    sudo chmod 750 "${BASE_DIR}/file_scripts/$prj"
done
```

### 4. 这种设计的强大之处 (Tradeoff)

- **真正的项目隔离**：即使项目 A 的 Agent 被黑了，它执行的 UID 是 ops-file-cms。这个 UID 在 OS 层面完全没有权限去动项目 B (ops-file-db) 的任何文件。
- **合规性**：在金融或高安全性场景下，这种“一项目一账号”是审计的加分项。
- **按需扩展**：你可以随时增加项目，只需要运行初始化脚本，而不需要修改核心的 PrivilegeBroker 类逻辑（因为它已经参数化了）。

### 5. 挑战：IntentClassifier 的压力

这种架构对 **IntentClassifier** 提出了更高的要求：

- 它不仅要判断命令是不是 file 类型。
- 它还要从用户请求中准确提取出 **“这是在操作哪个项目”**。
- **防越权检查**：LLM 必须确信，如果你请求的是项目 A，那么命令里的路径必须确实属于项目 A，不能挂羊头卖狗肉去删项目 B 的路径。

### 6. 现在的抉择 (Think Before Coding)

在 Week 4 的启动阶段，我们面临一个权衡：

- **选项 1 (简约优先)**：先维持目前的单 ops-file 架构，把“多租户”作为 Week 8 的目标。
- **选项 2 (前瞻设计)**：现在就把 project_id 引入 PrivilegeBroker 的接口，即便目前只有一个默认项目。

**根据 CLAUDE.md 的“简约至上”原则：**
我建议选 **选项 2 的轻量版**。即：接口支持 project_id，但默认值设为 "default"。这样我们既预留了扩展性，又不需要现在就去写复杂的多账号管理脚本。



# 是否显式指定执行账号

这个提议非常具有**前瞻性**，完全符合 CLAUDE.md 原则中的“表面权衡”和“目标驱动”。

通过在指令中显式包含 target_account（或 identity_context），我们能将 **“逻辑分类”** 与 **“物理账号”** 解耦。这为以后增加 ops-file-cms、ops-file-db 这种多租户架构打下了坚实基础。

以下是我们将该想法落地的具体方案：

### 1. 扩展 CommandRiskResult 数据结构

在 security/intent_classifier.py 中，我们为审查结果增加两个字段：

codePython

```
@dataclass
class CommandRiskResult:
    # ... 原有字段 (risk_level, reason 等)
    
    # 新增：显式指定账号类别和上下文
    category: Literal["read", "file", "service", "unknown"]
    context: str = "default"  # 默认为 default，未来可是 project_id
    
    @property
    def target_user(self) -> str:
        """组合成最终的 OS 账号名"""
        if self.context == "default":
            # 保持向后兼容：ops-reader, ops-file, ops-service
            return f"ops-{self.category}"
        return f"ops-{self.category}-{self.context}"
```

### 2. 改进 PrivilegeBroker.execute 接口

PrivilegeBroker 不再自己去猜测 cmd_type，而是直接接受审查员的建议，但保留**最后的否决权**。

codePython

```
def execute(
        self,
        cmd: str,
        risk_res: CommandRiskResult,  # 直接传入审查结果对象
        op_id: str,
        timeout: int = 30,
    ) -> ExecResult:
        
        # 1. 获取显式指定的账号
        user = risk_res.target_user
        
        # 2. 安全性二次校验 (Surgical Change)
        # 确保 LLM 没把一个 service 类型的命令投递给 reader 账号，或者反之
        self._validate_routing(risk_res)
        
        # 3. 动态确定脚本路径
        privilege = risk_res.category # "reader", "file", "service"
        script_path = self._write_script(cmd, privilege, op_id, context=risk_res.context)
        
        # ... 执行逻辑
```

### 3. 这种“显式指定”的三个好处：

1. **审计透明化**：在日志里我们可以清晰看到：*“LLM 建议使用 ops-file-projectA 执行此操作”*。这比模糊的“降级到 reader”更容易排查问题。
2. **权限最小化**：如果 Agent 同时管理 10 个项目，显式指定账号能确保它在操作项目 A 时，绝不可能误用到项目 B 的权限。
3. **应对 LLM 幻觉**：如果 LLM 幻觉了一个不存在的账号（如 ops-root），PrivilegeBroker 会在 execute 的第一步因为找不到账号或校验失败而拦截，这叫 **“输入验证”**。

### 4. 针对“多 ops-file”的未来演进（Think Before Coding）

当你决定增加项目时，你只需修改两处：

- **Prompt**：告诉 LLM，“如果你在处理 CMS 项目，请设置 context='cms'”。
- **Setup 脚本**：运行一个循环，创建 ops-file-cms 账号和对应的目录。
- **代码核心层**：**无需修改**。

既然决定加入显式账号字段，我们在 Week 4 的重构计划中增加一个关键动作：

我们需要让 LLM 在判定风险后，输出类似这样的结构：

codeJSON

```
  "context": "cms",

  "needs_human": false
}
```
